# Dataform Overview

How Dataform is used in this project, why it is a key technology, and how the same codebase runs both on Google BigQuery in the cloud and on DuckDB offline.

---

## Executive Summary

**For senior management:**

This programme replaces a 14-year legacy SQL Server regulatory-reporting estate — where the same reporting logic was written, tested and maintained separately for each of 17 regulated gaming markets — with **one codebase** built on **Google BigQuery and Dataform**. In the new platform, every market difference is *configuration data* and every piece of business logic exists *exactly once*. Adding a new jurisdiction becomes a configuration entry plus a small serialisation spec, not a new fork of the platform.

Dataform is the technology that makes this possible. It is Google Cloud's native framework for building SQL data pipelines on BigQuery: it turns our library of SQL-generating functions into a dependency-ordered build graph of ~200+ tables and views, executes them in the right order, and — critically for a regulated business — runs **~130+ named data-quality assertions, each traceable to a specific regulatory clause** (e.g. `rule_es_es_101`), before any file can reach a regulator. A failed assertion blocks the pipeline: **no bad data is ever submitted**.

Three properties of our Dataform implementation deserve board-level attention:

1. **Compliance is provable, not assumed.** Every regulatory constraint is a declarative rule with the regulator's clause reference as its identifier. A rule violation surfaces by name in the Dataform console and halts downstream submissions automatically.
2. **The entire pipeline runs offline in seconds, with zero cloud dependency or cost.** Because our SQL is generated by plain JavaScript functions rather than locked into Google's format, the identical logic executes against DuckDB (an embedded database) on a developer laptop. Every change is proven against seeded data, expected outputs, and deliberate "sabotage" tests *before* it is deployed — correctness is demonstrated before deploy, not discovered in production.
3. **Vendor exposure is contained.** Dataform is the production host, but the logic is portable JavaScript + SQL. The offline harness is standing proof that the codebase is not captive to GCP.

**Status:** the reporting core is fully built and verified for 9 of the 17 target markets (Malta, Spain, Denmark, Bulgaria, Greece, Netherlands, Germany, France, Portugal), with a documented deployment runbook for GCP. Production deployment, the SOAP submission service and the legacy parallel-run reconciliation are scoped and prototyped but not yet live.

## 1. What Dataform is, and why it was chosen

[Dataform](https://cloud.google.com/dataform) is Google Cloud's framework for managing SQL transformation pipelines in BigQuery. You describe *what* tables, views and data-quality assertions should exist and how they depend on each other; Dataform compiles that into a dependency graph (DAG), generates the DDL, executes everything in the correct order, and reports assertion failures as first-class, named objects.

It was chosen here because it directly answers the three problems that killed the legacy estate:

| Legacy problem | Dataform answer |
|---|---|
| 17 forks of the same stored-proc logic, one per market | One set of SQL-generating functions; Dataform fans them out per market from a single configuration file |
| Triggers and CHECK constraints scattered across databases, with no audit trail | Declarative assertions, named after regulatory clause ids, run on every pipeline execution and visible in one console |
| Correctness discovered in production | Compilation fails on invalid configuration; assertions block dependents; and the offline harness proves changes before deploy |

The guiding mantra, from `OVERVIEW.md`:

> **"Variance is data. Logic is singular. Correctness is proven before deploy."**

## 2. How Dataform is used in this project

There are two Dataform projects in the repository:

- **`dataform-example/`** — the fully-built reporting core: 9 markets, ~80 models, ~130 rule assertions, and the complete offline DuckDB harness. This is the target-state architecture.
- **`dataform-starter/`** — an earlier, thinner migration scaffold (2 markets). It still carries pieces the example has not yet absorbed: the legacy parallel-run **reconciliation layer** (`includes/recon.js`, `definitions/40_recon/`) with zero-breaks cutover gates, the Cloud Run **submission-service** skeleton, and the 4-phase strangler-fig `MIGRATION-GUIDE.md`.

### 2.1 The key design decision: no `.sqlx` files

A conventional Dataform project puts SQL in `.sqlx` files — a Dataform-proprietary format that only Dataform can execute. This project deliberately contains **none**. Instead it uses Dataform's **JavaScript API** (`publish()`, `declare()`, `assert()`), and every SQL statement is produced by a plain JavaScript builder function in `includes/`:

```js
// definitions/10_staging/staging.js — wiring only, no SQL
publish("stg_accounts", { type: "view", schema: "staging" })
  .query((ctx) => m.stgAccounts(ctx));   // SQL lives in includes/models.js
```

Because the SQL builders are ordinary pure functions `(ctx) => "SQL string"`, **any host can call them — Dataform is just one host**. This single decision is what enables offline execution (§4) and keeps the logic portable beyond GCP (§5).

### 2.2 The layered architecture

The deployable Dataform artifact is exactly three things: `workflow_settings.yaml` + `definitions/` + `includes/`.

**`includes/` — where all logic lives (once):**

| File | Responsibility |
|---|---|
| `jurisdictions.js` | The single source of truth for per-market variance: currency, timezone, tax model and (effective-dated) rates, submission cadence, report fields, player-protection policy, and the declarative rules — **data, never SQL** |
| `fields.js` | One SQL expression per submission field (e.g. `ggr`, `player_dni_hash`) |
| `models.js`, `queries.js`, `filters.js` | The SQL builders for staging/core/gaming models and submission queries |
| `rules.js` | The rule engine: turns declarative rule types (`not_null`, `unique`, `zero_when`, `no_unlicensed_games`, …) into violation queries |
| `validate.js` | Config pre-flight — an invalid configuration **fails compilation**, so a mistake can never silently ship |
| `exceptions.js`, `player_protection.js`, `extensions.js`, `effective_dating.js`, `nomenclature/` | Fault isolation/quarantine, responsible-gambling breach detectors, per-market bespoke attributes, versioned tax rates, sport/game code mapping |
| `dialect.js` | The BigQuery ⇄ DuckDB abstraction (§4.2) |

**`definitions/` — Dataform wiring only.** Numbered directories mirror the runtime DAG:

```
00_sources        cdc_* landing tables (Datastream CDC from SQL Server)
10_staging        dedupe append-only CDC (QUALIFY ROW_NUMBER … = 1)
15_reference      nomenclature mapping tables
20_core           jurisdiction-agnostic dims/facts (dim_customer_account, fct_bet_slip_lifecycle)
25_gaming         gaming activity, jackpot liability
30_submissions    per-market fan-out: submission_ready_*, tax_summary_*, periodic reports
35_player_protection / 40_exceptions
90_assertions     every declarative rule, compiled into a named Dataform assertion
```

The submissions layer simply loops over the jurisdiction config and publishes one incremental, date-partitioned `submission_ready_<market>` table per market into that market's own dataset (hard isolation, per-market IAM). Nine markets today; the tenth is a config entry.

### 2.3 Rules as regulator-traceable assertions

Every regulatory constraint is a declarative rule in `jurisdictions.js`, e.g. `{ id: "MT-103", type: "zero_when", field: "payout", when: "slip_status = 'VOIDED'" }`, where the id is the regulator's clause reference. `definitions/90_assertions/` compiles each rule into a Dataform assertion (`rule_mt_mt_103`) whose query **selects the violating rows — zero rows means compliant**. A red assertion in the Dataform UI points straight at the regulatory clause and blocks every dependent action, so a violation can never flow into a regulator file.

A second family — fault isolation (`includes/exceptions.js`) — is quarantine-first: failures route into `fct_exceptions` by class (data errors → quarantined, transient → retrying with backoff, incomplete sources → waiting on watermarks, RG breaches → held), and one hard-block assertion per market guarantees no quarantined entity reaches a submission.

## 3. How it runs on BigQuery (online / GCP)

In production (per the runbook `big-query-dataforms-howto.md`):

1. The pure workspace (`workflow_settings.yaml` + `definitions/` + `includes/`) is published to a Git-connected **Dataform repository** in GCP (`gcloud dataform repositories create reporting`).
2. Dataform compiles it — verified at **~200–280 actions across ~66 datasets** — resolving every `ctx.ref("…")` to a fully-qualified BigQuery name and a build edge, and generating the DDL (views, tables, incremental models with `partitionBy: "report_date"`, per-market datasets).
3. A **release config** recompiles hourly; a **workflow config** executes the full graph daily at 01:30 UTC — models in dependency order, then all assertions.
4. Source data arrives via **Datastream** CDC from SQL Server into the `cdc_*` landing tables (append-only, with `_op`/`_commit_ts` metadata); staging dedupes replays so at-least-once delivery is safe. The pipeline is fail-closed on `cdc_source_watermarks` — a market whose sources aren't confirmed complete ships nothing.
5. Downstream, **Cloud Workflows** triggers the translation layer (a Cloud Run job reusing `dataform-website/regulator_formats/`) **only after the Dataform run — models *and* assertions — reports SUCCEEDED**. That success signal *is* "data ready": a red assertion means no regulator XML is produced for anyone.

BigQuery is the default dialect: the Dataform path never calls `setDialect()`, so all builders emit BigQuery SQL as-is.

## 4. How it runs on DuckDB (offline)

### 4.1 The offline harness

`dataform-example/local/run.js` (`npm run local`) is a miniature stand-in that performs each of Dataform's jobs with **zero Google dependency** — the project's only npm dependency is `@duckdb/node-api`. It:

1. Switches the dialect to DuckDB (`setDialect("duckdb")`).
2. Loads deterministic seed data (`seed/data.js`) into an **in-memory DuckDB**.
3. Calls the **same** builder functions in `includes/` that Dataform calls, executing each as `CREATE OR REPLACE TABLE … AS <sql>` in hand-maintained dependency order (a flat `ctx.ref = (name) => name` replaces BigQuery qualification).
4. Runs **every rule assertion** — any returned row fails the run.
5. Checks integration **expectations** (`local/expectations.js`) — exact expected output rows per market.
6. Runs **negative tests**: deliberately corrupts copies of the data (duplicate slips, negative stakes, unlicensed games, wallet overspends, stake-limit breaches) and asserts that the corresponding rules fire.

The full cycle runs in seconds. `npm run check` (unit tests + demo compile + local pipeline + SQL emission) is the definition of done for any change. A fixed run timestamp makes retry/backoff behaviour deterministic offline. Two supporting scripts complete the story: `npm run dataform:compile` stages the pure workspace and runs a genuine `@dataform/cli` compile (proving it's a real Dataform project), and `npm run emit-sql` writes every generated statement to `dataform-sql/` in **both** dialects so SQL developers can read exactly what runs where.

Seeding is also single-sourced: `seed/data.js` generates both the DuckDB in-memory load and the BigQuery setup script (`seed/bigquery_setup.sql`) used for the cloud proof run.

### 4.2 The dialect layer

BigQuery and DuckDB SQL differ in a handful of constructs. `includes/dialect.js` is the **only** place engine-specific SQL is allowed — every builder calls dialect functions instead of inlining engine syntax, and each function has both implementations plus a unit test. Examples:

| Function | BigQuery | DuckDB |
|---|---|---|
| `sha256Hex(x)` | `TO_HEX(SHA256(CAST(x AS STRING)))` | `SHA256(CAST(x AS VARCHAR))` |
| `localDate(ts, tz)` | `DATE(ts, 'tz')` | `CAST(ts AT TIME ZONE 'tz' AS DATE)` |
| `normaliseText(x)` | `NORMALIZE(x, NFD)` + `REGEXP_REPLACE(…, r'\pM', '')` | `STRIP_ACCENTS(x)` |
| `regexpContains(x, p)` | `REGEXP_CONTAINS(x, r'p')` | `REGEXP_MATCHES(x, 'p')` |
| `dateTrunc('week', x)` | `DATE_TRUNC(x, WEEK(MONDAY))` | `DATE_TRUNC('week', x)` |

The cross-dialect diff of the emitted SQL shows `dialect.js` is the *only* difference between what the laptop runs and what production runs — which is what makes the offline results trustworthy evidence about production behaviour.

## 5. Dataform outside of GCP

Dataform is normally inseparable from BigQuery. This project deliberately loosens that coupling in three ways:

1. **The logic is host-independent.** Because all SQL lives in pure JavaScript functions rather than `.sqlx` files, Dataform is *a* host, not *the* host. The offline runner is a second host today; another warehouse (via a new dialect entry) would be a third.
2. **Dataform's open-source core is used without cloud infrastructure.** `local/dataform-compile.js` runs the standard `@dataform/cli` compile locally against a staged pure workspace — no GCP project, credentials or billing involved. Compilation (including config validation that fails the build on any mistake) is part of the desktop workflow.
3. **The surrounding demo stack is entirely local.** `dataform-website/` (a Flask app on a persistent DuckDB) plays the operator OLTP, producing data in the same `cdc_*` landing shapes Datastream would; a fictitious regulator SOAP endpoint (`safe.py`) receives the generated XML; and `regulator_formats/` — the translation layer that will run on Cloud Run in production — is stdlib-only Python, lifted (not rewritten) between the two environments. The complete end-to-end architecture is demonstrable by clicking, with no cloud account.

### Offline vs online at a glance

| Dataform's job | Online (BigQuery) | Offline (DuckDB) |
|---|---|---|
| Reference resolution | `ctx.ref` → fully-qualified name + DAG edge | flat `ref = (name) => name` |
| Execution order | derived automatically from the graph | hand-ordered plan array in `local/run.js` |
| DDL | views / tables / incrementals, datasets, partitioning | `CREATE OR REPLACE TABLE … AS` in one in-memory DB |
| Assertions | run post-build by Dataform, named in the console | same violation queries executed directly |
| Source data | Datastream CDC into `cdc_*` | deterministic seed from `seed/data.js` |
| Dialect | BigQuery (default) | `setDialect("duckdb")` |
| Cost / latency | cloud billing, scheduled daily | free, seconds, on every change |

## 6. Why this matters

- **Regulatory safety by construction.** Invalid configuration cannot compile; rule violations cannot reach a regulator; quarantined entities cannot enter a submission. Each guarantee is enforced by the pipeline, not by process discipline.
- **Marginal cost of a new market approaches configuration.** France, Portugal and the Netherlands were each added as jurisdiction entries + rules + a serialisation spec, exercising the same singular logic.
- **Fast, cheap, honest feedback.** Every change is proven against real SQL execution, expected outputs and adversarial negative tests before it leaves a laptop.
- **Contained vendor risk.** GCP is the production target, but the demonstrated portability (offline harness, dialect layer, host-independent builders) means the estate is not hostage to it.

**Honest caveats** (from `OVERVIEW.md`): the platform is not yet in production; the GCP deployment, SOAP submission service, full reconciliation/parallel-run layer and CI wiring are scoped and partially prototyped but not yet done. There is also a skills shift for the team — from writing SQL procedures to writing configuration and small SQL-generating functions (see `technology-skills-migration.md`).
